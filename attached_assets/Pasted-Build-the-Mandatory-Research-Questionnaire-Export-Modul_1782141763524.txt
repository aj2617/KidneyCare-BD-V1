## 1. PATIENT-SIDE: FORCED RESEARCH QUESTIONNAIRE

### 1.1 Trigger & Behaviour
- After a patient **logs in for the very first time**, immediately show a full‑screen overlay/page containing the research questionnaire.  
- The patient **cannot close or skip** the overlay; all other dashboard tabs/navigation are hidden or disabled until the survey is submitted.  
- Show a clear heading: “Complete Health Survey – Required for Better Care” (bilingual: “স্বাস্থ্য জরিপ সম্পূর্ণ করুন – ভালো যত্নের জন্য প্রয়োজনীয়”).  
- Provide a progress bar (6 sections).  
- Once submitted, the overlay disappears, the full dashboard unlocks, and the survey is marked as completed in the database.  
- The survey can later be **viewed and edited** from the Profile tab (no forced re‑submission, just an “Update Survey” button).  
- On subsequent logins, if already completed, no forced prompt appears.

### 1.2 Questionnaire Content (exact questions)
Use only these questions, grouped in sections. All text must be bilingual.

**Section 1 – Demographics**  
- Age (number, 18‑120, required)  
- Gender (Male / Female / Other, required)  
- Education (dropdown: No formal schooling / Primary / Secondary / Higher Secondary / Graduate or above, required)  
- Occupation (dropdown: Farmer, Housewife, Student, Service, Business, Unemployed, Other, required)  
- Residential area (Urban / Rural, required)  
- Monthly household income (BDT, number, optional)

**Section 2 – Medical History**  
- Diagnosed with high blood pressure? (Yes / No / Don’t know, required)  
- Diagnosed with diabetes? (Yes / No / Don’t know, required)  
- Family history of kidney disease? (Yes / No / Don’t know, required)  
- History of kidney stones? (Yes / No, required)  
- Previously diagnosed with any kidney‑related disease? (Yes / No; if Yes, specify type: CKD/Glomerulonephritis/PKD/Other, required)  
- Currently taking BP medication? (Yes / No, required)  
- Currently taking diabetes medication? (Yes / No, required)  
- Ever undergone dialysis? (Yes / No, required)

**Section 3 – Clinical Symptoms (last 3 months)**

- Persistent fatigue/weakness? (Yes / No, required)  
- Nausea/vomiting? (Yes / No, required)  
- Loss of appetite? (Yes / No, required)  
- Unexplained weight loss? (Yes / No, required)

**Section 4 – Lifestyle**

**Section 5 – Clinical Test Knowledge**

### 1.3 UI/UX Requirements
- Full‑screen overlay with a single scrollable white card (max‑width 480px, centered on larger screens).  
- Each section separated by a thin divider line and section title.  
- Questions as large radio buttons or dropdowns; touch targets ≥48px.  
- “Required” questions show a red asterisk; optional fields marked “(optional)”.  
- A prominent “Submit Survey” button at the bottom. While submitting, show a loading spinner.  
- On success, overlay fades out and a toast confirms completion.  
- The form must work within the PWA (offline access not required for this, but service worker should not break the page; online submission only).  
- Language toggle at top right of the overlay switches all text instantly.

### 1.4 Backend Requirements
- **New endpoint**: `POST /api/patient/survey`  
- Payload: JSON object containing all survey answers.  
- The backend verifies the patient’s JWT, checks if a survey already exists (if yes, update; if no, create).  
- Store survey data in a new Mongoose collection `PatientSurveys` with fields: `patientId` (ref to User), `responses` (mixed), `completedAt`, `updatedAt`.  
- Also update the Patient model with a boolean `surveyCompleted: true/false` for quick dashboard check.  
- On successful save, return `{ success: true }`.  

- **Get survey status**: `GET /api/patient/survey/status` returns `{ completed: boolean }` – used by frontend to decide whether to show the overlay.

- **Get survey data for editing**: `GET /api/patient/survey` returns the full response object if exists.

## 2. ADMIN PANEL: RESEARCH DATA EXPORT

### 2.1 Access & UI
- Add a **“Research Export”** card/button in the Admin **More** tab (or a dedicated tab if space).  
- Label: “Research Data Export (Anonymized)” / “গবেষণা তথ্য রপ্তানি (বেনামী)”.  
- Opens a minimal modal/bottom sheet with options:  
  - District filter (dropdown, all districts)  
  - Date range (optional, to filter by survey completion date)  
  - Export format: CSV (default).  
- A single “Generate & Download” button.

### 2.2 Data Composition
The exported CSV must merge **survey responses** with the patient’s key **clinical CKD data** from the dashboard. Each row = one patient. Columns to include (exactly):

- **Anonymized ID** (a hash of patient’s MongoDB _id, e.g., `CRC32` or random number assigned once, stored in patient record to keep consistent)  
- **Survey Data**: Age, Gender, Education, Occupation, Urban/Rural, Income, all medical history answers, symptom answers, lifestyle, test knowledge, awareness.  
- **Clinical Data** (latest values at time of export):  
  - Latest eGFR values (MDRD, Cockcroft‑Gault, CKD‑EPI) from GFRRecords  
  - CKD Stage (derived from latest eGFR)  
  - Latest Risk Score from risk engine  
  - Latest Blood Pressure (systolic/diastolic) from VitalsLog  
  - Latest Blood Sugar (fasting/random) from VitalsLog  
  - Latest Creatinine from VitalsLog  
  - Latest Weight from VitalsLog  
  - Total number of vitals logs submitted  
  - Date of first vitals log  
  - Date of last vitals log  

- **No personally identifiable information** (name, email, phone, exact address) – only anonymized ID and district.

### 2.3 Backend Export Logic
- **Endpoint**: `GET /api/admin/research-export`  
- Query params: `district`, `startDate`, `endDate`.  
- Server queries `PatientSurveys` joined with `Patients`, `GFRRecords`, `VitalsLog` (using aggregation pipeline).  
- Generates CSV string with headers as above, all string values properly escaped.  
- Sends the CSV as a downloadable file with `Content-Disposition: attachment; filename=kidneycare_research_export.csv`.  
- Only Admin role can access this endpoint.

### 2.4 Data Integrity & Ethics
- The export must be **strictly anonymized**. No direct patient IDs, names, or contact info.  
- If a patient has not completed the survey, they are excluded from the export.  
- All optional fields (income, BP, creatinine) must appear as empty strings if not provided.
