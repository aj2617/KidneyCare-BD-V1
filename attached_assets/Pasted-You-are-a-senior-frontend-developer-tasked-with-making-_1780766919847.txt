You are a senior frontend developer tasked with making the entire "KidneyCare BD" Progressive Web App **fully responsive** on every device — mobile phones (360px–428px), tablets (768px–1024px), and desktops (1280px and above). The application already has all features, dashboards, and a bottom navigation bar for mobile. Your job is to refine and guarantee perfect responsiveness everywhere.

## 1. OBJECTIVE
Ensure that every page, component, and dashboard (Patient, Doctor, CHW, Admin) adapts flawlessly to any screen size, is touch‑friendly on mobile, and maintains a clean, usable layout on desktop. No horizontal scrolling, no overlapping elements, no text cut‑off.

## 2. WHAT YOU MUST DO
- Audit the current UI across the four dashboards.
- Fix any layout breakages found at the following breakpoints:
  - `min-width: 360px` (small mobile)
  - `min-width: 768px` (tablet)
  - `min-width: 1280px` (desktop)
- Implement or improve the following responsiveness rules:

### 2.1 Global Layout
- Use a **mobile‑first** CSS approach (Tailwind’s default). All base styles target mobile; override with `md:` and `lg:` prefixes for larger screens.
- On screens < `768px`:
  - Each dashboard uses a **bottom navigation bar** (already present, but verify it does not overlap content).
  - Main content area takes `height: calc(100vh - 56px)` (or the actual bottom bar height).
  - Forms, charts, and lists stack vertically, taking full width.
- On screens ≥ `768px`:
  - Replace bottom nav with a **fixed left sidebar** (or top nav for some roles if specified). Ensure smooth transition.
  - Sidebar width: `240px` on tablet, `280px` on desktop.
  - Main content area fills remaining space with proper padding.
- Sidebar and bottom navigation must never appear simultaneously.

### 2.2 Typography & Readability
- Base font size: `16px` on mobile, `16px` on tablet, `16px` on desktop (do not scale down below 16px for readability).
- Headings scale using relative units (`rem`) or Tailwind’s responsive text classes (`text-lg md:text-xl`).
- Bengali text must always use `Noto Sans Bengali` and be clearly legible at all sizes.

### 2.3 Touch & Interaction
- All interactive elements (buttons, links, form fields) must have a minimum touch target of `48x48px` on mobile (enforced via Tailwind `min-h-[48px] min-w-[48px]` or equivalent).
- Form inputs must use appropriate `inputmode` attributes (e.g., `numeric`, `tel`) to trigger the correct mobile keyboard.
- No hover-dependent critical functionality; use `active` states instead or in addition.

### 2.4 Data Tables & Lists
- Tables that overflow on small screens must be wrapped in a scrollable container (`overflow-x-auto`), or converted to card‑based layouts on mobile.
- Patient lists, alert lists, and user management tables must be swipeable and use compact card designs on mobile.

### 2.5 Charts & Maps
- All Recharts/Chart.js charts must use `ResponsiveContainer` with `width="100%"` and `aspect={…}` or fixed height that scales.
- The Leaflet.js heatmap must fill its container; on mobile, it should occupy the top half of the screen; on desktop, it should expand to fill the available area.
- Map controls (zoom, layer toggles) must be large enough to tap.

### 2.6 Modals, Bottom Sheets & Drawers
- On mobile, use bottom sheets (or full‑screen modals) for forms (e.g., log vitals, issue prescription). These must slide up from the bottom and cover most of the screen.
- On desktop, the same actions can appear as side panels or centered modals.
- Ensure all modals are scrollable if content exceeds viewport height.

### 2.7 PWA-Specific Responsiveness
- The installed PWA must respect the `standalone` display mode: no browser chrome, so bottom nav must not be hidden by OS navigation bars. Use `safe-area-inset-bottom` padding.
- Offline indicator banner must be positioned fixed and responsive (full width, thin banner).
- The "Add to Home Screen" prompt must be non‑intrusive and centred on mobile, dismissible on desktop.

## 4. IMPLEMENTATION RULES
- Use **Tailwind CSS** utility classes exclusively for layout, spacing, and responsive behaviour.
- Do not write custom CSS unless absolutely necessary; if so, keep it minimal and scoped.
- Use `flex`, `grid`, and responsive padding/margin utilities.
- Test every page using Chrome DevTools device toolbar at these exact widths: **375px (iPhone SE), 414px (iPhone 11), 768px (iPad), 1280px (laptop)**.
- Fix all instances of:
  - Horizontal scrollbars
  - Text truncated or overflowing
  - Buttons too small to tap
  - Modals that are not scrollable
  - Map that does not resize
- After fixes, re‑verify PWA installation and offline behaviour.
